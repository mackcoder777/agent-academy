=== MOST IMPORTANT CHANGE — READ FIRST ===

THE MINIMUM VIABLE AGENT PRINCIPLE is now the core UX law.

The intake is NOT a fixed 10-step form. It is an adaptive conversation
that ends the moment the agent has enough to launch.

Step count is determined by agent complexity:
- Simple (summarize, extract, classify): 3 exchanges, under 5 minutes
- Medium (structured output, template filling): 3-4 exchanges, under 8 minutes
- Complex (multi-doc, cross-reference, judgment): 5 exchanges, under 12 minutes

THE RULE: If the answer wouldn't change what the agent does, don't ask the question.

Nothing blocks launch except:
- No concept defined
- No output defined

Everything else is optional and improves through dashboard use.

=== REDESIGNED INTAKE STEPS ===

PRE-STEP (before anything):
Single field: "What do you want to automate?"
Claude classifies: industry, workflow_type, agent_class, complexity
Returns required_steps array — only these steps are shown
No dropdown. User never picks an industry.

STEP: CONCEPT (always required)
"What should your agent do?"
Coaching surfaces 1-2 gaps with injectable options specific to this workflow.
If description is complete enough, skip gaps and move forward immediately.

STEP: WHAT DOES IT READ? (always required)
Runtime inputs only. What the user provides each time.
SILENT ROUTING: If user mentions contracts/specs/SOPs → tagged for step 6 silently.
If user mentions past examples → tagged for dashboard improvement queue.

STEP: WHAT DOES IT PRODUCE? (always required)
Outcome-phrased options, not architecture decisions.
If output is templated → template step added to flow.
If output is unstructured → no template step needed.

STEP: STANDING CONTEXT (required for medium/complex, optional for simple)
7 named categories for construction:
1. Contracts and Scope — subcontract, prime contract, scope of work
2. Specifications — project specs, addenda, bulletins
3. Drawings — drawing sets, sheet indexes, ASIs
4. Company Standards — SOPs, response templates, internal guidelines
5. Approved Lists — approved subs, approved products, preferred vendors
6. Codes and Regulations — OSHA, building codes, local AHJ
7. Historical Reference — past RFI logs, past CO decisions, resolved disputes

Non-construction industries get equivalent categories generated by Claude.
Multiple selections always work. Selecting one never hides others.
"I'll add this later from my dashboard" always available.

STEP: OUTPUT TEMPLATE (required only if output is structured/templated)
Path A: User uploads → best-effort automated mapping → launch → refine in dashboard
Path B: No template → app generates one based on industry standard → user accepts or skips
Path C: Skip → best-effort output → dashboard queues template upload
NEVER blocks launch regardless of path.

STEP: HUMAN GATES (required only for complex agents making consequential decisions)
Toggle-based — not text fields, not open-ended questions.
Simple agents: this step is skipped entirely, smart defaults applied silently.

STEP: NAME AND LAUNCH (always last)
Plain English summary of what was built.
Non-blocking sandbox test: shows what agent does, does not block launch.
"This looks right → Launch" or "One thing is off → fix it → launch"

=== TEMPLATE MAPPING ===

DURING BUILD: Best-effort automated mapping. Agent launches on this.
No cell-by-cell review. No blocking before launch.

AFTER LAUNCH (dashboard mapping session):
User chats with LLM in plain English to correct mapping.
"That column is the RFI number, not the date."
LLM updates. Agent incorporates immediately.

The build session NEVER becomes a mapping session.

=== SANDBOX ===

Active from the first step where output can be shown.
User uploads real file at any point during build.
Output improves visibly as steps are completed.
"WHAT WOULD IMPROVE THIS" section drives them forward.
Desktop: right panel. Mobile: bottom sheet.

=== BLUEPRINT OUTPUT ===

Blueprint compiles to deployable JSON spec (not prose).
Claude Code reads the JSON and builds:
  agent.py, state.py, tools.py, retrieval.py, formatter.py,
  gates.py, failures.py, logger.py, requirements.txt, Dockerfile

Hosted on Railway. Each agent gets its own isolated container.
Endpoint: https://agents.agentacademy.app/[agent_uuid]/run

=== PRICING ===

FREE_RUN_LIMIT = environment variable, default 1.
Change to 5 or 10 during testing without redeploying.
Run 1: free. Run 2+: requires Stripe subscription ($199/month).

=== DASHBOARD (post-launch) ===

Run history: every run as a card (expand, download, flag)
Performance panel: success rate, avg confidence, avg cost, sparklines
Improvement queue: failures ranked by impact, one-click fixes
Template mapping session: post-launch LLM chat for precision
Revision chat: LLM with full blueprint + run history context
Step editor: reopen any intake step, save → redeploy → compare baseline

=== ARCHITECTURE (always reliable underneath) ===

Regardless of how simple the UX:
- State persisted before returning (no data loss)
- Every run logged with full metadata (no silent failures)
- Failure types classified immediately
- Human gates enforced when configured
- pgvector index maintained even if document library starts empty
- Retry transient failures 3 times before notifying
- Graceful degradation on standing context miss
